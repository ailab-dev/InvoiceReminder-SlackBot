## 4. 型定義

### 4.1 `types/intern-salary.ts`

```typescript
// 経費1件分
export type ExpenseItem = {
  name: string;    // 費目名（例: "移動費"）
  amount: number;  // 金額（円）
};

// インターン生のプロフィール（不変情報・更新可）
export type InternProfile = {
  slack_user_id: string;
  name: string;
  address: string;          // 請求書の請求者住所
  phone: string;
  bank_info: string;        // "〇〇銀行 〇〇支店 普通 1234567"
  unit_price: number;       // 円/時
  expense_names: string[];  // 前回使った費目名（最大3件・次回自動入力用）
  updated_at: string;       // ISO 8601
};

// 月次提出データ（確定後）
export type InternSalarySubmission = {
  id: string;                   // UUID v4
  intern_id: string;            // Slack User ID
  intern_name: string;
  intern_address: string;
  month: string;                // "YYYY-MM"
  unit_price: number;
  working_hours: number;
  expenses: ExpenseItem[];      // 経費一覧（空行を除いた最大3件）
  total_salary: number;         // floor(unit_price × working_hours)
  total_expense: number;        // expenses の amount 合計
  subtotal: number;             // total_salary + total_expense（税抜）
  tax_amount: number;           // round(subtotal × 0.10)
  total_amount: number;         // subtotal + tax_amount（税込）
  bank_info: string;
  payment_due: string;          // "YYYY-MM-DD"
  invoice_number: string;       // "INV-{YYYY}-{MM}-{SlackUserID}"
  status: "submitted" | "paid";
  submitted_at: string;         // ISO 8601
  paid_at: string | null;
};

// モーダル private_metadata に詰める一時確認データ（Pendingキー廃止につき）
export type InternSalaryDraft = Omit<
  InternSalarySubmission,
  "id" | "status" | "submitted_at" | "paid_at"
> & {
  phone: string;  // プロフィール更新用（submissionには不要なので分離）
};
```

> **Note**: `InternSalaryPending` 型と Redis の `intern:salary:pending:*` キーは廃止。
> モーダル2ページ構成により、確認データは Slack の `private_metadata`（最大3000文字）で保持する。

## 5. データ設計（Upstash Redis）

### 5.1 Redis キー設計

| キー | 型 | 内容 |
|---|---|---|
| `intern:profile:{slack_user_id}` | String (JSON) | `InternProfile` |
| `intern:known_users` | Set | 過去に提出した Slack User ID |
| `intern:salary:{id}` | String (JSON) | `InternSalarySubmission` |
| `intern:salary:months:{YYYY-MM}` | Set | その月の提出 ID 集合 |
| `intern:salary:submitted_users:{YYYY-MM}` | Set | その月に提出済みの User ID 集合 |

### 5.2 計算ルール

```
total_salary   = floor(unit_price × working_hours)
total_expense  = sum(expenses[].amount)
subtotal       = total_salary + total_expense
tax_amount     = round(subtotal × 0.10)
total_amount   = subtotal + tax_amount
payment_due    = 来月の末日
invoice_number = "INV-{YYYY}-{MM}-{SlackUserID}"
```

**PDF レイアウト方針**:
- `@react-pdf/renderer` の `Document` / `Page` / `View` / `Text` コンポーネントで構築
- フォント: デフォルトフォント（日本語表示のため Noto Sans JP を埋め込む）
- 用紙: A4縦
- ファイル名: `invoice_{YYYYMM}_{intern_name}.pdf`（呼び出し元で設定）

> **日本語フォントの埋め込み**:
> `@react-pdf/renderer` のデフォルトフォントは日本語非対応のため、
> `Font.register()` で Noto Sans JP を登録する。
> フォントファイルは Google Fonts CDN から取得するか、プロジェクトに同梱する。

5. フォルダをサービスアカウントのメールアドレスに「編集者」として共有
6. フォルダの URL から ID を取得（`drive.google.com/drive/folders/{FOLDER_ID}`）

#### ケース E: モーダル1ページ目送信（`callback_id: "intern_salary_modal"`）

**変更点**: Pending保存・確認DM送信を廃止。`response_action: "push"` で確認画面をプッシュする。

**処理フロー**:
```
1. 署名検証
2. 入力値を取得・バリデーション（数値チェック）
3. 金額計算
   total_salary  = floor(unit_price × working_hours)
   total_expense = expense_transport + expense_travel + expense_ai
   subtotal      = total_salary + total_expense
   tax_amount    = round(subtotal × 0.10)
   total_amount  = subtotal + tax_amount
   payment_due   = 来月末日
   invoice_number = "INV-{YYYY}-{MM}-{user_id}"
4. InternSalaryDraft を JSON化 → モーダルの private_metadata に詰める
5. response_action: "push" で確認モーダルを返す（2ページ目）
```

**バリデーションエラーの返し方**（変更なし）:
```typescript
return NextResponse.json({
  response_action: "errors",
  errors: { unit_price_block: "正の整数を入力してください" }
});
```

**確認モーダル（2ページ目）の定義**:
**経費パース処理**（`intern_salary_modal` 送信時）:
```typescript
// モーダルの3行分の経費を配列に変換（両方空の行は除外）
const expenses: ExpenseItem[] = [1, 2, 3].flatMap((i) => {
  const name = values[`expense_${i}_name_block`]?.[`expense_${i}_name_input`]?.value?.trim() ?? "";
  const amountRaw = values[`expense_${i}_amount_block`]?.[`expense_${i}_amount_input`]?.value?.trim() ?? "";
  if (!name && !amountRaw) return [];  // 両方空 → スキップ
  // name あり amountRaw なし（またはその逆）→ バリデーションエラー（上でチェック済み）
  return [{ name, amount: Number(amountRaw) }];
});
const total_expense = expenses.reduce((sum, e) => sum + e.amount, 0);
```

#### ケース F: 確認モーダル2ページ目送信（`callback_id: "intern_salary_confirm_modal"`）

**変更点**: `confirm_intern_salary` ボタン廃止。このケースが確定処理の本体となる。
PDF生成・配布・Google Driveアップロードをここで実行する。

**処理フロー**:
```
1. 署名検証
2. payload.view.private_metadata から InternSalaryDraft を取得
3. user_id = payload.user.id
4. UUID生成 → InternSalarySubmission を構築
5. saveSubmission(submission)
6. saveInternProfile(profile)   // プロフィール更新（住所・expense_names含む）
   // expense_names = submission.expenses.map(e => e.name)（最大3件・空文字で埋める）
7. HTTP 200 を返す（モーダルを閉じる）← ここで3秒制限を回避

--- 以下は非同期で実行 ---
8. generateInvoicePdf(invoiceData) → Buffer生成
9. 並列実行:
   a. Slack files.uploadV2 → インターン生DM（pdfBuffer）
   b. Slack files.uploadV2 → INTERN_SALARY_CHANNEL_ID（pdfBuffer）
   c. uploadToDrive(pdfBuffer, fileName, GOOGLE_DRIVE_FOLDER_ID)
10. インターン生DM: 完了通知メッセージ送信
11. チャンネル投稿: 提出通知メッセージ送信（佐藤さん向け）
```

> **3秒制限の回避**: `view_submission` には先に `return new NextResponse(null, { status: 200 })` を返す。
> その後の処理は `waitUntil` や `Promise` で非同期実行する。
> Vercel の Hobby プランでは `waitUntil` が使えないため、`fetch` を `await` せずに投げる方法で代替する。

## 10. エラーハンドリング方針

| 状況 | 対応 |
|---|---|
| 署名検証失敗 | 403 を返す |
| CRON_SECRET 不一致 | 401 を返す |
| 数値入力が不正 | モーダルバリデーションエラー（`response_action: "errors"`） |
| `private_metadata` のパース失敗 | 200 を返してスキップ（冪等性確保） |
| PDF生成失敗 | ログ出力・Slack通知は継続（PDFなしで通知） |
| Google Drive アップロード失敗 | ログ出力・継続（SlackへのPDF送信は完了させる） |
| Slack API エラー | ログ出力・Cron は次の処理を継続 |
